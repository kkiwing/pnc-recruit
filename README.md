# P&C 채용관리 (P&C Recruitment Manager)

## 1. 프로젝트 개요

P&C(인사)팀의 채용 프로세스를 관리하기 위한 웹 애플리케이션 **프로토타입**입니다. 채용 공고 등록부터 지원자 접수, 전형 단계 진행 상황, 면접 일정까지 한 화면에서 관리하는 흐름을 다룹니다.

현재 버전은 실제 백엔드 없이 동작하는 **순수 프론트엔드 인메모리 프로토타입**입니다. 앱을 새로고침하면 모든 데이터가 `src/data/` 아래의 더미 데이터로 초기화됩니다.

## 2. 목적

이 저장소는 프로덕션에 배포할 서비스가 아니라, **테크팀이 실제 개발에 착수할 수 있도록 화면 흐름과 데이터 구조를 미리 검증해보는 기획용 프로토타입**입니다. P&C팀이 원하는 채용 관리 화면/기능/데이터 모델을 먼저 프론트엔드로 구현해보고, 이를 바탕으로 테크팀과 요구사항을 정렬한 뒤 실제 백엔드가 있는 서비스로 재구축하는 것을 목표로 합니다.

## 3. 주요 기능

- **공고 관리**: 채용 공고 등록/수정/삭제, 상태(진행중/마감/준비중) 관리, 공고별 지원 현황 요약
- **지원자 관리**: 지원자 등록/수정/삭제, 공고별 지원자 목록 조회, 이름·팀·이메일 검색
- **채용 프로세스 관리**: 면접 일정 캘린더 및 시간표, 별도관리(미응시·중도포기 등) 대상 지정 및 사유 기록, 별도관리 대상 목록으로 복원
- **전형 단계/상태 관리**: 인성검사 안내→등록→결과, 자사양식 안내→제출, 면접 안내→결과로 이어지는 7단계 전형 상태를 지원자별로 인라인 관리 (대기/필요/완료/합격/불합격)
- **지원자 메모/이력 관리**: 지원자별 특이사항 메모, 첨부파일 업로드(현재는 브라우저 세션 내 임시 저장), 등록일·수정일 기록

## 4. 기술 스택

- **프레임워크**: React 18 + Vite 5 + TypeScript
- **UI**: shadcn/ui (Radix 기반), Tailwind CSS, lucide-react
- **라우팅**: react-router-dom
- **상태 관리**: React Context + `useState` (인메모리, 백엔드 연동 없음)
- **테스트**: Vitest + Testing Library (단위), Playwright(e2e 설정은 있으나 현재 미사용)
- **패키지 매니저**: npm

> `@tanstack/react-query`, `react-hook-form` + `zod`, `recharts`는 설치되어 있지만 현재 화면에서는 사용하지 않습니다. 실제 백엔드 연동 시 데이터 페칭/폼 검증/차트 용도로 활용을 고려해볼 수 있습니다.

## 5. 로컬 실행 방법

```bash
npm install
npm run dev
```

기본적으로 `http://localhost:8080`에서 실행됩니다 (포트가 사용 중이면 다른 포트로 자동 전환).

기타 스크립트:

```bash
npm run build     # 프로덕션 빌드
npm run lint       # ESLint 검사
npm run test       # Vitest 단위 테스트 실행
```

## 6. 환경변수 설정 위치

**현재는 별도의 환경변수가 필요하지 않습니다.** 이전에는 Supabase 연동을 위한 `.env`가 있었으나, 순수 인메모리 프로토타입으로 전환하면서 제거했습니다.

향후 실제 백엔드를 연동하게 되면, 프로젝트 루트에 `.env` 파일을 생성해 `VITE_` 접두사가 붙은 값들을 정의하면 Vite가 자동으로 읽어들입니다. `.env`는 `.gitignore`에 포함되어 있으므로 커밋되지 않습니다 — API 키나 시크릿을 코드에 직접 넣지 마세요.

## 7. 현재 프로토타입의 한계

- **데이터가 영속되지 않음**: 모든 상태가 브라우저 메모리에만 존재하며, 새로고침 시 `src/data/dummyApplicants.ts` / `dummyJobPostings.ts`의 초기값으로 리셋됩니다.
- **인증/권한 없음**: 로그인 기능이나 담당자별 권한 구분이 전혀 없습니다.
- **파일 첨부 미영속화**: 지원자 첨부파일은 `URL.createObjectURL`로 생성한 임시 blob URL만 사용하며, 새로고침하면 사라집니다.
- **면접 평가표(스코어카드) 없음**: 면접 결과는 합격/불합격 이진값만 기록하며, 면접관별 평가 항목·점수 기능은 없습니다.
- **오퍼/입사 관리 없음**: 최종 합격 이후의 처우 협의, 입사일 확정 등 온보딩 프로세스는 다루지 않습니다.
- **알림/이메일 발송 없음**: "안내" 단계들은 상태값만 기록할 뿐 실제 지원자에게 메일/문자를 보내지 않습니다.
- **변경 이력(감사 로그) 없음**: 전형 단계 상태는 마지막 값으로 덮어써지며, 누가 언제 무엇을 변경했는지 추적할 수 없습니다.
- **테스트 커버리지 낮음**: Vitest 예시 테스트 1개만 존재하며, Playwright e2e 설정은 있으나 실제 spec 파일과 연동 패키지 설정이 정리되지 않은 상태입니다.
- **일부 화면 라우팅 누락**: `src/pages/ApplicantList.tsx`, `src/pages/SeparateManagement.tsx`는 구현되어 있으나 라우트/사이드바 네비게이션에 아직 연결되지 않았습니다.

## 8. 향후 테크팀 구축 시 참고할 점

- **데이터 모델의 출발점**: `src/types/applicant.ts`, `src/types/jobPosting.ts`에 정의된 프론트엔드 도메인 타입이 실제 서버 스키마 설계 시 좋은 출발점이 될 수 있습니다. 단, `JobPosting.stages`/`Applicant.stageRecords`처럼 여러 단계를 배열/중첩 객체로 다루는 구조는 실서비스에서는 단계별 이력 추적이 가능한 정규화된 테이블(예: `recruitment_stages`, 상태 변경 이력 테이블)로 재설계하는 것을 권장합니다. 합불 판정 단계 식별(`Stage.stageType`)과 완료/합격/불합격 상태 식별(`StageStatus.isCompletion`/`isPass`/`isFail`)처럼 "의미"를 명시적 필드로 표현하는 패턴은 순서나 이름 문자열에 의존하는 것보다 안전하므로, 서버 스키마에서도 동일한 방향을 유지하는 것을 권장합니다.
- **CRUD 로직은 전면 교체 필요**: `src/context/ApplicantContext.tsx`, `src/context/JobPostingContext.tsx`의 등록/수정/삭제 함수는 현재 로컬 상태만 다루는 자리표시자입니다. 실제 서버 연동 시 API 호출 기반으로 완전히 다시 구현해야 합니다. 지원자 번호(`no`) 채번, 공고 삭제 시 소속 지원자 cascade 삭제 등은 클라이언트에서 흉내만 낸 로직이므로, 서버(DB 트랜잭션/외래키 제약)에서 제대로 처리해야 합니다.
- **UI 컴포넌트는 재사용 가능**: `src/components/`의 화면 구성과 shadcn/ui 기반 컴포넌트들은 데이터 소스만 교체하면 그대로 활용 가능하도록 설계되어 있습니다.
- **인증/권한, 파일 스토리지, 알림 발송**은 이 프로토타입에 전혀 없으므로 요구사항 정의 단계부터 새로 설계가 필요합니다.
