# 의사결정 로그

## 2026-07-20 — 롤플레이 검증 반영: 발송 직전 개별 수정, 자동 발송 옵트인 원칙, 기록 아이콘을 "기록 존재" 기준으로

**배경**: 담당자 롤플레이 검증에서 나온 개선 요구를 반영했다. (1) 발송 섹션의 미리보기가 읽기 전용이라 특정 지원자에게만 문구를 고쳐 보내려면 전체 지원자 공용인 프로세스 관리의 템플릿을 건드려야 했다. (2) 자동 발송이 기본 ON이어서 담당자가 인지하지 못한 발송 기록이 생길 수 있었다. (3) 시계/발송 아이콘이 "현재 상태가 hasDateInput인가" 기준이라, 상태가 진행완료 등으로 넘어가면 아이콘이 사라져 과거 기록에 접근할 방법이 없었다.

**결정 1 — 발송 직전 개별 수정, 실제 발송 내용을 기록에 보존**: 발송 섹션의 제목/본문을 편집 가능한 입력으로 바꿨다. 변수 치환본으로 채워져 시작하고(안 고치면 날짜 변경을 따라가는 템플릿 추적 상태, 한 글자라도 고치면 고정), "템플릿으로 되돌리기"로 리셋할 수 있다. 발송 시 실제 보낸 제목/본문을 `meta.send.subject/body`로 보존하고, 재발송 모달은 지난 발송본에서 시작한다(수정 유지). 자동 발송은 템플릿 치환본이 그대로 기록된다. 템플릿(프로세스 관리)의 역할은 "기본형 관리"로 명확해졌다 — 개별 수정은 템플릿 원본에 영향을 주지 않는다.

**결정 2 — 자동 발송은 어디서든 기본 OFF(옵트인 원칙)**: 발송은 대외 행위이므로 켜는 것은 담당자의 명시적 선택이어야 한다. 기본 프리셋 4단계·더미 공고·새로 추가되는 단계 모두 자동 발송 OFF로 시작한다(제목/본문 템플릿과 채널 기본값(이메일+문자)은 채워둔다 — 수동 발송은 바로 쓸 수 있게). 새 단계의 기본 제목은 `{{전형단계명}} 안내`로, 변수 그대로 저장해 발송 시점에 단계명이 자동 반영된다.

**결정 3 — 기록 아이콘은 "기록 존재" 기준(히스토리 접근성)**: 시계 아이콘을 "hasDateInput OR meta 기록 존재"면 항상 노출로 바꿨다(기록 없으면 흐리게=입력 유도, 있으면 진하게 — 메모 아이콘과 같은 문법). 발송 기록(meta.send)이 있으면 메일 아이콘을 상태와 무관하게 항상 표시한다. 적용 범위는 시계 아이콘이 나오는 모든 곳(지원자 목록/별도관리 테이블 — 별도관리의 "당시 진행 단계" 셀 포함 — /지원자 상세 전형 현황/스테퍼). **잠금은 변경 차단이지 열람 차단이 아니다**: 최종 결과로 잠긴 지원자도 기록이 있으면 시계를 클릭해 열람 전용 모달(모든 입력·발송 비활성, 닫기만)로 확인할 수 있다. 이와 함께 프리셋/새 단계의 모든 상태에 날짜+메모(hasDateInput)를 기본 체크했다 — 어느 상태로 바뀌든 기록을 남길 수 있는 것이 기본값이고, 끄는 것이 담당자의 선택이다.

**함께 반영**: `{{회사명}}` 치환 상수를 P&C → ACG로 정정(대외 발송 주체는 회사명이지 인사팀 브랜딩이 아님). 발송 모달은 세 진입점(목록 시계/상세 시계/상태 변경) 모두 동일한 `CompletionDateModal` 컴포넌트임을 확인했다(통일 작업 불필요).

**재검토가 필요한 항목**: 재발송이 지난 발송본을 덮어쓰는 구조(이력 없음)와 회사명 설정화면 부재는 2026-07-18 항목의 재검토 사항이 그대로 유효하다. 별도관리 지원자의 기록 열람은 추가됐지만, 별도관리 상태에서 기록을 "수정"할 경로는 여전히 없다(복귀 후 수정) — 의도된 제약.

## 2026-07-18 — 발송 버튼은 프로세스 관리가 아닌 지원자 모달에: 개별 발송 도입과 토글 의미 재정의

**배경**: 자동 발송 설정 UI는 있었지만 발송이라는 행위 자체가 어디에도 없었다(설정 저장만 가능). 발송 기능을 프로토타입 수준(기록만)으로라도 넣으면서, "발송 버튼을 어디에 둘 것인가"를 정해야 했다.

**결정 1 — 발송 버튼은 프로세스 관리가 아니라 개별 지원자의 날짜 입력 모달에 둔다**: 프로세스 관리 화면에 발송 버튼을 두면 "이 단계의 지원자 전원에게 발송"이라는 의미가 되기 쉬운데, 이는 오발송(잘못된 템플릿·잘못된 대상 다수에게 일괄 발송) 위험이 크고, 발송 시점에 필요한 개별 맥락(그 지원자에게 안내할 날짜·시간)이 프로세스 관리 화면에는 없다. 반면 날짜 입력 모달은 담당자가 특정 지원자의 특정 단계에 날짜를 입력하는 순간이라, 변수 치환된 메시지를 그 자리에서 미리보기로 확인하고 발송할 수 있다 — 발송은 "지원자 한 명 단위"의 행위라는 원칙. 프로세스 관리는 템플릿(메시지 원본)의 소유처로만 남긴다.

**결정 2 — 토글의 의미를 "자동 발송 사용"으로 좁히고 템플릿 폼은 항상 노출**: 기존에는 토글이 꺼져 있으면 템플릿 폼 전체가 접혀서 "기능이 사라졌다"는 오해(2026-07-17 항목)까지 낳았다. 템플릿은 자동·수동 발송이 공용으로 쓰는 자원이므로 토글과 묶일 이유가 없다. 폼은 항상 보여주고, 토글은 "해당 상태로 변경 확정 시 자동 발송되는가"만 결정한다. 아코디언 라벨도 "자동 발송 설정" → "발송 메시지 설정"으로 바꿨다.

**결정 3 — 프로토타입은 발송 기록만 남긴다**: 실제 메일/SMS 발송 없이 `StageRecord.meta.send = { sentAt, channels, auto? }`만 기록하고 "발송됨 날짜 · 채널"로 표시한다. 재발송은 기록 갱신(덮어쓰기)이다. 자동 발송은 `hasDateInput` 상태로의 변경을 모달에서 확정하는 시점에만 `auto: true`로 기록된다 — meta 재편집 경로에서는 재발송되지 않는다(중복 발송 방지). 변수 치환 유틸(`src/lib/messageTemplate.ts`)은 값 없는 변수를 토큰 그대로 남겨 미확정 항목이 미리보기에서 드러나게 했다. 상세 정책은 functional-spec 2.15.

**재검토가 필요한 항목**:
- **실제 발송 연동(메일/SMS API)은 향후 과제** — 사내 메일 시스템 연동 방식이 정해지면, 현재의 발송 기록 지점(모달의 발송 버튼, 자동 발송 확정 지점)이 그대로 실제 발송 호출 지점이 된다.
- 재발송 시 기록을 덮어쓰므로 발송 이력이 남지 않는다 — 실제 연동 시 이력 테이블로 확장 필요.
- `{{회사명}}`은 코드 상수다 — 실제 서비스에서는 설정 화면으로 이전 필요. (2026-07-18 롤플레이 검증에서 발송 주체가 P&C(인사팀)가 아니라 회사명 ACG여야 한다는 피드백을 받아 상수 값을 'ACG'로 정정했다. 팀 브랜딩과 대외 발송 주체는 다른 값이라는 확인 — 설정 화면으로 옮길 때도 이 구분을 유지해야 한다.)
- 발송 기록이 `meta`에 있어서 그 단계를 날짜 입력 없는 상태로 바꾸면 날짜/메모와 함께 발송 기록도 사라진다(기존 meta 수명 규칙을 그대로 따름). 발송 기록을 상태 전환과 무관하게 보존해야 한다면 저장 위치를 분리해야 한다.

## 2026-07-17 — "자동 발송 설정 기능이 사라졌다"는 제보 조사 결과: 회귀 아님, 기본값 미설정이 원인 + 기본 템플릿 세팅

**배경**: 프로세스 관리의 "자동 발송 설정" 아코디언을 열면 토글과 "설정 저장" 버튼만 보이고, 이메일/문자 채널, 제목, 내용, 변수 삽입 칩이 사라진 것 같다는 제보가 있었다.

**조사 결과 — 코드 회귀가 아니다**: `git log --all -- src/components/process/AutoSendPanel.tsx`로 확인한 결과, 이 컴포넌트는 최초 생성 커밋(`b21fa8b`, 2026-07-13) 이후 **단 한 번도 수정된 적이 없다** — 채널 체크박스, 제목 입력, 내용 textarea, 변수 삽입 칩({{지원자명}} 등 7종), 저장 버튼, 안내 문구가 전부 최초 구현 그대로 남아 있다. `ProcessManagement.tsx`도 여전히 이 컴포넌트를 정상적으로 import·렌더링한다. 실제로 브라우저에서 토글을 켜보면 이메일/문자·제목·내용·변수 삽입 칩이 모두 정상적으로 나타나는 것을 확인했다.

보이는 현상의 진짜 원인은 컴포넌트 설계 자체다 — 처음부터 `{draft.enabled && (...)}`로 켜져 있을 때만 나머지 필드를 보여주도록 만들어져 있었고(최초 커밋부터 이랬음, 회귀가 아님), **더미 프리셋·더미 공고의 모든 단계가 `autoSend`를 아예 갖고 있지 않았거나 `enabled: false`였다.** 즉 "기능이 사라진 것"이 아니라 "꺼진 상태를 접어서 보여주는 원래 동작"이 우연히 "기능이 없어진 것"처럼 보였을 뿐이다. `git log -S`로 찾아봐도 `"[{{회사명}}] 인성검사 안내"` 같은 기본 템플릿 문자열이 이 저장소 역사에 존재한 적이 없다 — "다시 세팅"이라는 표현과 달리, 더미 데이터에 기본 템플릿이 채워져 있던 적은 처음부터 없었던 것으로 보인다.

**결정한 방향**: 회귀가 없으므로 코드를 "복구"할 것은 없었다. 대신 요청받은 실질적인 목표(아코디언을 열면 바로 완성된 예시가 보이는 것)를 위해, 기본 프로세스 프리셋(`createDefaultStages()`)과 더미 공고 프리셋(`dummyJobPostings.ts`의 `defaultStagesFor`) 양쪽의 4개 단계 모두에 `autoSend: { enabled: true, channels: [...], title, body }` 기본 템플릿을 새로 채워 넣었다 — 인성검사/자사양식/최종은 이메일만, 면접은 이메일+문자(SMS)를 기본으로 켰다(면접은 당일 리마인드 성격이 강해 SMS가 실무적으로 더 잦다는 통상적 관행을 반영). 본문에는 해당 단계 성격에 맞는 변수({{지원자명}}/{{회사명}}/{{포지션명}}/{{링크}}, 면접은 추가로 {{면접일시}}/{{면접장소}})를 넣었다. 프리셋과 개별 공고(job-01) 양쪽에서 "자동 발송 설정 (사용 중)" 라벨과 채워진 필드가 정상 노출되는 것을 확인했다.

**재검토가 필요한 항목**: 없음. `AutoSendPanel.tsx` 자체는 손대지 않았고(회귀가 없었으므로), 더미 데이터에 기본값만 추가했다.

## 2026-07-17 — 프로세스 관리 단계 카드도 동일한 드래그앤드롭 패턴으로 통일 + 재사용 가능한 형태로 굳힘

**배경**: 상태값 관리 모달의 순서 변경을 실제 드래그앤드롭으로 바꾼 직후, 같은 화면(프로세스 관리)의 **단계 카드** 순서 변경은 여전히 위/아래 화살표 버튼이었다. 같은 화면 안에 두 가지 다른 순서 변경 UX가 공존하는 것을 피하기 위해 동일한 패턴으로 통일했다.

**결정한 방향**:
- `ProcessManagement.tsx`의 단계 카드에 상태값 모달과 동일한 도트 드래그 핸들을 추가하고, 위/아래 화살표 버튼(`moveStage`)을 제거했다. 핸들 판정(mousedown ref), 반투명+삽입선 피드백, `reorderStage`(splice 기반 재정렬)까지 전부 동일한 패턴을 그대로 재사용했다.
- **추가로 발견해 함께 고친 문제**: `dragover`/`drop` 핸들러가 `draggingIndex`(state)를 직접 참조하고 있었는데, `dragstart` 직후 매우 빠르게 연달아 발생하는 `dragover`가 아직 커밋되지 않은 리렌더 이전의 낡은 클로저(`draggingIndex === null`)를 참조해 `preventDefault`를 놓치는 경우가 실제로 있었다(자동화 테스트 중 재현). state와 별개로 항상 최신값을 동기적으로 반영하는 `draggingIndexRef`를 추가해 `dragover`/`drop` 판단은 이 ref로만 하도록 두 파일(`StageStatusModal.tsx`, `ProcessManagement.tsx`) 모두 수정했다. state(`draggingIndex`)는 계속 시각적 표시(반투명/삽입선) 용도로만 쓴다.
- 아코디언(자동 발송 설정)이 열려 있는 카드를 드래그해도, 열려 있던 다른 카드가 함께 딸려가도 문제없이 동작하는 것을 확인했다 — `key={stage.id}`로 각 카드 서브트리(아코디언 열림 상태 포함)가 단계 자신의 정체성을 따라가므로, 순서가 바뀌어도 "어느 카드가 열려 있었는지"는 그대로 유지된다.
- 프리셋 편집 모드와 실제 공고 편집 모드 양쪽에서 동일하게 동작하는 것을 확인했다.

**검증 방법과 한계**: 실제 마우스 드래그(mousedown→mousemove→mouseup) 시뮬레이션은 이번에도 시도했지만, 이 프로토타입 검증에 쓰는 브라우저 자동화 툴이 native HTML5 드래그의 `dragover`/`drop` 연속 발생을 안정적으로 재현하지 못했다(같은 좌표·같은 코드로 반복해도 결과가 들쭉날쭉 — dragstart만 발생하거나, 중간에 dragend로 끊기는 경우가 잦았다. Puppeteer/Playwright 커뮤니티에도 잘 알려진, CDP의 단순 마우스 이벤트 시퀀스로는 네이티브 드래그 세션을 끝까지 유도하기 어렵다는 한계와 일치). 대신 실제 마우스가 핸들을 누르면 발생하는 것과 동일한 이벤트 시퀀스(핸들에 진짜 `mousedown` 디스패치 → 카드에 `dragstart`/대상 카드에 `dragover`·`drop`·`dragend`, 각 이벤트의 `target`도 실제 네이티브 동작과 동일하게 맞춤)를 직접 재현해, 재정렬·아코디언 상태 보존·프리셋/공고 양쪽 동작을 전부 확인했다. 이 방식은 "로직이 올바른지"는 확실히 검증하지만 "실제 마우스 제스처 자체가 브라우저에서 안정적으로 감지되는지"까지 보장하지는 않는다 — 상태값 모달 쪽은 이전에 실제 마우스 시뮬레이션으로 최소 1회 전체 사이클(dragstart→dragover→drop→dragend)이 완주되는 것을 이미 확인했고, 이번 단계 카드 구현은 그 코드를 기계적으로 그대로 재사용한 것이라 동일하게 동작할 것으로 판단했다.

## 2026-07-17 — 상태값 관리 모달: "단계 종료"를 isDefault와 대칭인 위치 기반 자동 플래그로 전환, 순서 변경을 실제 드래그앤드롭으로

- 순서 변경 UI를 화살표 버튼(이전 커밋에서 도트+팝오버 메뉴로 한 차례 교체했던 것)에서 **실제 드래그앤드롭**으로 바꿨다. 도트 모양 핸들(`GripVertical`)에서 시작한 드래그만 인정해, 이름 입력창 안에서 텍스트를 드래그로 선택하는 동작과 충돌하지 않게 했다. 드래그 중인 행은 반투명하게, 놓일 위치는 파란 삽입선으로 표시한다. 재정렬 자체는 `react-beautiful-dnd`의 표준 `reorder` 구현과 동일한 splice 기반 함수를 그대로 썼다.

**버그와 수정(실제 브라우저 확인 후)**: 처음 구현은 "핸들에서 시작한 드래그만 인정"하는 조건을 `dragstart` 핸들러 안에서 `event.target.closest('[data-drag-handle]')`로 검사하는 방식이었다. 합성 `DragEvent`를 직접 `dispatchEvent`로 쏴서 검증했을 때는 통과했지만, 실제 마우스로 조작해보니 핸들을 정확히 잡고 끌어도 드래그 자체가 시작되지 않았다. 원인은 HTML5 Drag and Drop 스펙의 잘 알려지지 않은 동작 — **`dragstart` 이벤트의 `event.target`은 마우스가 눌린 실제 하위 요소가 아니라, 항상 `draggable`이 걸린 조상 요소(드래그 소스) 자신**이라는 점이었다(다른 대부분의 DOM 이벤트와 다르다). 즉 `event.target`은 언제나 행(`draggable` div) 자신이었고, `data-drag-handle`은 그 행의 자식에 붙어 있었으니 `target.closest(...)`는 항상 실패해 매번 `preventDefault`로 드래그가 취소되고 있었다. 앞서 합성 이벤트로 "검증됐다"고 판단한 것은 `element.dispatchEvent()`가 호출 대상 요소 자체를 `target`으로 만들어버려, 실제 네이티브 동작과 다른 조건을 테스트한 것이었다(검증 자체가 잘못됐던 사례). 수정은 핸들의 `onMouseDown`/`onMouseUp`으로 "핸들이 눌려있는 상태"를 `ref`에 별도로 기록해두고, `dragstart`에서 `event.target` 대신 그 `ref`를 확인하는 방식으로 바꿨다. 이번에는 실제 마우스 드래그 시뮬레이션(합성 이벤트 직접 발생이 아니라 진짜 mousedown→mousemove→mouseup 시퀀스)으로 재정렬과 시작/종료 배지 위치 이동, 그리고 핸들이 아닌 곳(이름 입력창)에서 드래그를 시작하면 취소되고 텍스트 선택은 그대로 되는 것까지 확인했다.

**재검토가 필요한 항목**: 없음. `isCompletion`을 소비하는 다른 코드(`ProcessManagement.tsx`의 "(단계종료)" 표시, `isStageCompleted` 등)는 필드 자체의 의미가 그대로이므로 수정이 필요 없었다.

- 고정 목록이 없어졌으니 사유 필터 셀렉트는 의미가 없어져 제거했고, 대신 검색어가 사유 텍스트도 훑도록 넓혔습니다.

**배경 (최종 결과 지정 시 전형 잠금)**: `finalResult`를 지정한 뒤에도 목록/파이프라인/상세 화면에서 전형 단계·상태를 계속 바꿀 수 있어서, "최종 결과"와 "지금 전형이 어디까지 왔는지"가 서로 어긋나 보일 수 있었습니다.

**결정한 방향 (전형 잠금)**: `finalResult`가 `null`이 아닌 지원자는 전형 조작 UI(목록의 전형/상태 셀렉트, 파이프라인의 드래그와 "다른 단계로 이동" 메뉴, 상세 페이지의 날짜·메모 재입력 버튼)를 전부 비활성화하고 "최종 결과가 지정되어 전형이 종료된 지원자입니다"를 안내합니다. 최종 결과 배지 자체는 잠그지 않아 "판정 해제"로 되돌릴 길은 남겨뒀고, 해제하는 즉시 잠금도 자동으로 풀립니다. 세 화면이 같은 로직을 반복 구현하지 않도록 공용 `LockedTooltip` 컴포넌트로 통일했습니다.

자세한 내용은 `docs/functional-spec.md` [2.7](../docs/functional-spec.md#27-별도관리-이동복귀와-separatestageid-스냅샷-규칙), [2.13](../docs/functional-spec.md#213-최종-결과finalresult-지정-시-전형-조작-ui-잠금)에 정리했습니다.

## 2026-07-15 — 단계별 합불 판정 폐지, 최종 결과(finalResult)로 일원화

**배경**: 단계별 합불 플래그 도입 후 UI 복잡도 증가로 폐지, 최종 결과(finalResult)로 일원화. 프로토타입 검증 과정에서의 방향 전환. 구현된 상태값 관리 화면을 실제로 보니 한 단계 안에 "시작/단계종료/날짜+메모/합격으로/불합격으로"라는 다섯 개 개념이 동시에 존재해, 담당자가 상태 하나를 만들 때마다 이 조합을 다 이해해야 했습니다.

**결정한 방향**:
- `StageStatus.isPass`/`isFail`, `Stage.stageType`(`'normal' | 'result'`)을 완전히 제거했습니다. 상태값 관리 화면의 "구분" 셀렉트, "합격으로"/"불합격으로" 지정 UI도 함께 제거했습니다.
- "단계별로 합불을 판정한다"는 개념 자체를 폐지했습니다. "합격"/"불합격"이라는 이름의 상태는 이제 그냥 이름과 색만 가진 일반 상태로 존재할 수 있습니다(진행 상황 표시 용도일 뿐, 집계에는 쓰이지 않습니다). 기본 프리셋의 인성검사/면접 단계는 이 형태로 유지합니다.
- 합격자/불합격자 집계(대시보드 "최종 합격" 카드, 공고 카드의 합격 뱃지, 면접 완료 테이블의 결과 컬럼 등 전부)는 이미 있던 `Applicant.finalResult` 단일 기준으로만 판단하도록 통일했습니다 — 애초에 stage 기반 합불 판정과 finalResult 두 경로가 공존했던 것 자체가 복잡도의 원인이었습니다.
- "단계 추가" 시에도 더 이상 합불 타입 분기가 없습니다. 항상 대기(시작)/진행중/완료(단계종료) 3상태로 생성됩니다(`progressStatuses()` 단일 빌더).
- "단계종료"(`isCompletion`) 체크는 이제 완전한 배타 선택입니다. 예전에는 합격/불합격 두 상태가 동시에 `isCompletion:true`를 가질 수 있도록 예외를 뒀지만, 그 예외의 존재 이유(합불 판정)가 사라졌으므로 한 단계에 종료 상태는 항상 하나만 유지됩니다.
- 상태값 관리 모달에 남은 것: 이름, 색상, 단계종료 체크박스, 날짜+메모 체크박스, 삭제, 순서 이동, 목록 맨 위 상태의 "시작" 뱃지.

**재검토가 필요한 항목**: 없음 — 이번 결정으로 2026-07-13 항목("고정 7단계 전형 구조를 공고별 커스터마이징 가능한 구조로 전환")에서 도입했던 `stageType`/`isPass`/`isFail` 필드 언급은 더 이상 유효하지 않습니다. 그 항목은 기록 보존을 위해 원문 그대로 남겨두되, 실제 구현 기준은 이 항목을 따릅니다.

## 2026-07-15 — 전형 구조를 "단계 세분화"에서 "국면 + 상태"로 재편

**배경**: 기존 7단계 구조는 인성검사 하나를 안내→공고등록→결과 3단계로, 자사양식/면접도 각각 안내→제출(결과) 2~3단계로 잘게 쪼개 놓았습니다. 이렇게 잘게 쪼갠 단계들은 실제로는 "전 단계가 완료되면 다음 단계가 대기 상태로 시작한다"는 의미가 반복될 뿐이었고, 완료 처리 방식(`completionForm`)도 단계 속성으로 박혀 있어 "이 단계의 완료 = 날짜를 입력받는 단 하나의 전환"이라는 암묵적 가정을 강제했습니다. 특별 채용처럼 정해진 절차를 다 거치지 않는 예외를 다룰 방법도 없었습니다(합불 판정이 항상 "마지막 합불 판정 단계"의 상태로 결정됐으므로).

**결정한 방향** — "구조는 단순하게, 예외는 메모로":
- 전형 단계를 세부 절차가 아니라 전형의 **국면** 단위로 통합했습니다. 기본 프리셋은 이제 인성검사/자사양식/면접/최종 4단계이고, 예전에 단계로 나뉘어 있던 세부 진행(안내됨/등록됨/결과 나옴 등)은 그 국면 하나의 **상태값**으로 흡수했습니다.
- 완료 처리 시 날짜를 입력받을지는 단계 속성이 아니라 **상태 속성**(`StageStatus.hasDateInput`)으로 옮겼습니다. 어떤 상태로 전환하든(꼭 "완료"가 아니어도) 그 상태가 날짜 입력을 요구하는지 독립적으로 정할 수 있습니다.
- "시작 상태"는 상태 목록의 첫 번째 항목과 항상 자동으로 일치하도록 만들어(수동 지정 UI 삭제), "이 단계에 오면 자동으로 갖는 상태가 뭔지" 헷갈릴 여지를 없앴습니다. "단계 종료"(`isCompletion`, 예전 이름 "완료")는 한 단계 안에 여러 상태가 동시에 가질 수 있게 풀어서(예: 합격/불합격 둘 다), "이 단계는 끝났다"를 판정하는 로직이 특정 하나의 상태를 찾는 대신 현재 상태 자체의 플래그만 보면 되게 했습니다.
- **최종 합불 판정을 전형 진행 상태에서 완전히 분리**해 `Applicant.finalResult`(`{ result, note?, decidedAt }`)라는 별도 필드로 뺐습니다. 전형 단계가 어디까지 진행됐는지와 무관하게 지정할 수 있어서, 특별 채용 같은 예외를 새로운 상태값·단계를 늘리지 않고 메모 한 줄로 처리할 수 있습니다.

**재검토가 필요한 항목**: `finalResult`를 지정할 수 있는 시점이나 권한에 지금은 아무 제약이 없습니다(전형이 전혀 진행되지 않은 지원자도 바로 합격 처리 가능). 실 서비스에서는 이 부분에 업무 규칙이 필요할 수 있습니다. 자세한 정책은 `docs/functional-spec.md` 2장(특히 2.3, 2.4, 2.11, 2.12)에 정리했습니다.

## 2026-07-14 — 하드코딩된 기본 7단계를 관리 가능한 "기본 프로세스 프리셋"으로 승격

**배경**: 새 공고를 등록하면 항상 `createDefaultStages()`가 만든 고정된 7단계가 그대로 적용됐습니다. 실제로는 조직의 기본 채용 프로세스 자체가 시간이 지나며 바뀔 수 있는데, 이를 바꾸려면 코드를 수정해야 했습니다.

**결정한 방향**:
- 프로세스 관리 화면에서 공고를 선택하지 않은 상태(첫 진입)를 "기본 프리셋 편집" 화면으로 삼았습니다. 공고 셀렉트 최상단에 "기본 프리셋" 항목을 추가했습니다.

- 프리셋은 `ProcessPresetContext`에 순수 인메모리 상태로 유지됩니다. **새로고침하면 `createDefaultStages()`로 초기화됩니다** — 다른 화면의 데이터와 마찬가지로 이 프로토타입 단계의 한계이며, 실제 구축 시에는 프리셋도 서버에 영속화해야 합니다.

**재검토가 필요한 항목**: 프리셋을 여러 개(직군별 등) 두는 요구가 생기면 지금의 단일 프리셋 구조를 확장해야 합니다.

## 2026-07-13 — 디자인 시스템 도입, 라이트 모노크롬 톤으로 UI 통일

**배경**: 화면이 늘어나면서 카드/뱃지/셀렉트 스타일이 화면마다 제각각으로 구현되고 있었고, 색상도 시맨틱 토큰이 아닌 하드코딩된 값에 의존하는 경우가 많았습니다.

**결정한 방향**: `docs/DESIGN_SYSTEM.md`에 색상·그림자·라운딩·타이포그래피 토큰 기준을 정리하고, 이후 모든 UI 작업은 이 문서를 기준으로 합니다(`CLAUDE.md`에도 명시). 사용자가 직접 지정하는 단계 상태 색상처럼 토큰으로 고정할 수 없는 경우에는 배경색만 값으로 받고 텍스트 색은 대비를 계산해 자동으로 정하는 방식(`StatusBadge`)으로 처리합니다.

## 2026-07-13 — 고정 7단계 전형 구조를 공고별 커스터마이징 가능한 구조로 전환

**배경**: 실제 채용에서는 공고 유형에 따라 전형 단계 수와 이름, 조건이 다릅니다(예: 특정 공고에만 임원 면접이 추가되는 경우). 하드코딩된 7단계 구조로는 이런 차이를 반영할 수 없었습니다.

**결정한 방향**:
- 공고(`JobPosting`)가 자신의 전형 단계(`stages`)를 직접 소유하고, 지원자(`Applicant`)는 단계별 진행 기록(`stageRecords`)을 갖는 구조로 전환했습니다.
- 프로세스 관리 화면에서 담당자가 단계 추가/삭제/순서 변경, 상태값(이름/색상) 구성을 직접 할 수 있습니다.
- "합불을 판정하는 단계"와 "완료/합격/불합격을 의미하는 상태"를 배열 순서나 이름 문자열로 암묵적으로 추정하지 않고, `Stage.stageType`과 `StageStatus.isCompletion`/`isPass`/`isFail` 같은 명시적 필드로 판별하도록 했습니다. (순서 기반 추정은 단계를 재배치하거나 새로 추가하는 순간 오판정으로 이어지는 것을 확인했습니다.)

**재검토가 필요한 항목**: 이 구조를 실제 서버 스키마로 옮길 때는 배열/중첩 객체 대신 단계별 이력 추적이 가능한 정규화된 테이블 설계를 권장합니다(자세한 내용은 README "향후 테크팀 구축 시 참고할 점" 참고).

## 2026-07-13 — 지원자 목록에 목록형/파이프라인(칸반) 뷰 병행 제공

**배경**: 지원자 수가 늘어날수록 테이블 스크롤만으로는 전형 단계별 분포를 파악하기 어렵습니다.

**결정한 방향**: 목록형 테이블과 파이프라인(칸반) 뷰를 토글로 전환할 수 있게 하고, 검색·필터·정렬 상태는 두 뷰가 공유하도록 했습니다. 파이프라인 뷰에서는 카드를 드래그하거나 메뉴로 다른 단계로 옮길 수 있습니다.

## 2026-07-13 — 순수 인메모리 프로토타입으로 전환

**배경**: 이 단계의 목표는 실제 서비스 운영이 아니라 테크팀 구축 전 화면·데이터 흐름 검증입니다. 외부 연동(백엔드/DB)을 프로토타입 단계부터 유지하는 것은 검증 속도 대비 복잡도가 컸습니다.

**결정한 방향**: 모든 상태를 React Context + 더미 시드 데이터로 관리하는 순수 프론트엔드 구조로 정리했습니다. 새로고침 시 데이터가 초기화되는 것은 이 단계에서 감수하는 트레이드오프입니다.

**재검토가 필요한 항목**: 실제 구축 시 어떤 백엔드/DB, 인증 방식을 쓸지는 이 프로토타입의 범위 밖이며 테크팀과 별도로 정해야 합니다.
